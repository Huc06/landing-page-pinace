## What each layer owns

| Layer | Owns | Doesn't own |
| --- | --- | --- |
| **Wallet** | Owner key (encrypted local), pool ownership UI, policy management, agent sign approvals. | Trading logic, indexer state. |
| **Agent app** | Intent → PTB conversion (chat, MCP, bot). Holds the agent keypair (or asks the wallet to). | Owner key. Funds. |
| **SDK** | PTB builders (`@pinace/core`), low-level Move bindings (`@pinace/contracts-sdk`), read client. Pure functions, no I/O of its own. | Storage, signing. |
| **Indexer** | Denormalised read API + SSE stream over chain events. | Settlement authority — chain is. |
| **Sui chain** | Source of truth. `BalancePool` + `Policy<W,C>` objects + their `propose_action → settle` PTB pipeline. | Everything else. |

## End-to-end swap flow

When a user types *"Swap 0.05 WSUI to WUSDC"* in Fenik:

  participant I as Indexer
  U->>F: "Swap 0.05 WSUI → WUSDC"
  F->>I: GET /agents/:id (read policy state)
  I-->>F: spending_limit { spent_in_window: 0.6 / 2 }
  F->>F: pre-flight checkSwapAllowed (off-chain mirror)
  F->>C: getQuote (DeepBook RPC)
  F->>F: build PTB (propose → prove → take → swap → return → settle)
  F->>W: pinace:signAndExecuteAsAgent(tx)
  W->>U: ① Sign approval popup
  U->>W: Sign
  W->>W: ② Executing progress (Sign → Broadcast → Finalise)
  W->>C: signAndExecuteTransaction(agent key)
  C-->>W: digest
  W-->>F: { digest }
  C-->>I: events (ActionProposed, FlowAuthorized, FlowCompleted, ActionSettled)
  I-->>W: SSE pool_deposit / pool_withdraw
  I-->>F: SSE action_settled
  F-->>U: "Done — view on Suiscan"`} />

Every numbered step is verifiable. The user's owner key was used
**zero times** for the swap — they signed the policy attach once,
and the agent does the rest, bounded by Move.

## The hot-potato — Pinace's killer pattern

Everything above hangs on one Move idiom. `propose_action` mints a
**`Request` hot-potato** — a struct with no `key`, no `store`, no
`drop`. Once it exists, the PTB **must** consume it via `settle`
or the whole transaction reverts.

<Mermaid chart={`flowchart LR
  P[propose_action] -->|mint Request| R((Request))
  R --> POL[policy.prove]
  POL --> R
  R --> TK[take_coin]
  TK --> V[venue call]
  V --> RC[return_coin]
  RC --> R
  R --> S[settle]
  S -->|consume Request| OK[committed]
  S -->|missing receipt| RV[whole tx reverts]`} />

That single requirement forces every policy `prove`, every venue
call, and the funds in/out of the pool to live in the **same**
atomic PTB.

Why this is the killer feature:

- **No "half-done" state.** EVM lets `approve` and `transferFrom`
  sit in different txs; Pinace can't — they're literally the same
  PTB.
- **Inline enforcement.** Spending limit + slippage cap + every
  custom policy run in the same atomic step as the spend.
- **Composable across venues.** A future Cetus or Scallop wrapper
  drops into the same shape; existing policies gate it for free.

## Two guarantees this buys you

- **Atomicity.** The propose → prove → settle pipeline is one PTB.
  If the spending-limit assert fails, the DeepBook swap doesn't
  happen. If DeepBook returns less than `min_amount_out`, nothing
  moves. There is no "half-done" state to recover from.

- **Revocability without a sweep.** The owner can burn the agent's
  authority with one signed `revoke_agent` call. The next
  `propose_action` from that address aborts with `E_REVOKED`. No
  per-token approval to clean up — see
  [Revocation](/docs/concepts/revocation).
