> **Other agent wallets hand the AI your keys. Pinace hands it a budget.**

Pinace is a non-custodial wallet that lets AI agents act on a user's
funds **without ever holding their keys**. The owner deposits into an
on-chain `BalancePool`, attaches one or more policies (spending limits,
slippage caps, time windows), and any wallet-standard dApp can drive
an agent that swaps inside those exact bounds.

## The killer: hot-potato settlement

Pinace's superpower is the **hot-potato PTB**. Every agent action —
swap, lend, stake, anything — runs as one atomic transaction
shaped like:

```text
propose_action → policy.prove → take_coin → venue call → return_coin → settle
```

Each step lives in the **same PTB**. The `Request` object is a
hot-potato (no `key`, no `drop`) — once `propose_action` mints one,
the PTB **must** consume it via `settle` or the whole transaction
aborts.

What this guarantees:

- **Atomicity** — there is no "half-done" swap on chain. Either
  every policy passes + the venue fills + funds return to the pool,
  or **nothing moves**.
- **Inline enforcement** — policy `prove` runs in the same tx as
  the venue call. The spending limit can't be checked-then-spent
  past; the slippage cap can't be bypassed by a second tx.
- **Composability** — N policies and 1 venue stack into one PTB.
  Add a new venue, the same policies still gate it without changes.

This is why Pinace doesn't need EVM-style allowances or trusted
sequencers — Move enforces the bounds in the same atomic step as
the spend. See [Hot-potato PTB](/docs/concepts/hot-potato-ptb) for
the full pattern.

## How it fits together

<Mermaid chart={`flowchart LR
  USER["Owner (Pinace Wallet)"]

  DEX["DeepBook v3 / Cetus / ..."]
  USER -- "1. deposit" --> POOL
  USER -- "2. attach policy" --> POLICY
  USER -- "3. connect" --> AGENT

  POLICY -- "5. prove" --> POOL
  POOL -- "6. swap" --> DEX
  DEX -- "7. return_coin" --> POOL`} />

Every step is a verifiable on-chain transaction. The agent never holds
the owner's key. Policies are checked inside the **same** PTB as the
swap — if anything breaks (spending limit, wrong coin, slippage), the
whole transaction reverts.

## Two ways to integrate

**You're building an agent or chat app** → start with [Quickstart](/docs/quickstart)
and the [Wallet Standard features](/docs/wallet) `pinace:connectAgent`
+ `pinace:signAndExecuteAsAgent`. Two RPCs and dapp-kit gives you
everything else.

**You're a venue (DEX, lending, yield)** → read [Adding a venue](/docs/integrations#adding-a-venue)
and the [PTB action shape](/docs/sdk). Your team writes a Move
action handler; Pinace gates it with the user's existing policies.
